

## عن هالمشروع



**الفكرة العامة للمشروع:** رح تبني بيئة AWS آمنة ومنظمة (IAM)، بعدين تشغّل سيرفر ويب (EC2) بطريقة صحيحة وآمنة، بعدين تتعلم تدير التخزين تبعه (Storage/AMI)، وأخيرًا تخليه **Highly Available و Scalable** عن طريق Load Balancer و Auto Scaling Group. يعني بالنهاية رح يكون عندك تطبيق ويب حقيقي شغال بطريقة احترافية تقريبًا.

**طريقة الاستخدام:** كل خطوة إلها 4 أجزاء ثابتة:
- 🔧 **شو بتعمل** — التعليمات (اعمل هيك بالضبط)
- 📝 **شو سويت** — ملخص لما عملته
- ⚙️ **وظيفته وميزاته** — ليش هاد الشي موجود وشو بيحلّه
- 🧠 **أشياء لازم تعرفها** — نقاط تخص فهمك العميق (وبتفيدك بالامتحان كمان)

## المرحلة 0: التحضير والحماية من التكاليف

### الخطوة 0.1: تفعيل IAM access للـ Billing + عمل Budget

🔧 **شو بتعمل:**
1. سجّل دخول بحساب الـ Root (مو IAM user).
2. روح على اسم الحساب أعلى يمين → Accounts → انزل لتحت لغاية "IAM user and role access to billing information" وفعّلها.
3. روح على Billing and Cost Management → Budgets → أنشئ:
   - **Zero Spend Budget** (ينبهك أول ما تصرف 1 سنت)
   - **Monthly Cost Budget** بحد $10 مع تنبيهات عند 85% و100% من الصرف الفعلي، و100% من الصرف المتوقع (Forecasted)
4. حط إيميلك بالمكانين.

📝 **شو سويت:** فعّلت صلاحية وصول الـ IAM users للفواتير، وعملت نظام تنبيهات تلقائي لو صرفت مصاري بالغلط.

⚙️ **وظيفته وميزاته:** AWS ما بوقف الخدمات تلقائيًا لو تجاوزت الفري تير — لازم إنت تراقب. الـ Budget هو خط الدفاع الأول: بيبعثلك إيميل فورًا، مش بآخر الشهر لما توصلك الفاتورة.

🧠 **أشياء لازم تعرفها:**
- افتراضيًا الـ IAM users (حتى لو Admin) **ما بيقدروا يشوفوا الـ Billing** إلا إذا الـ Root فعّلها.
- Free Tier Dashboard (بالـ sidebar) بيوريك استهلاكك الحالي والمتوقع مقابل حدود الفري تير.
- لو فاتورتك طلعت غريبة، روح Bills → اختار الشهر → Charges by Service — هاد بيوريك بالضبط أي خدمة كلفتك قديش.

## المرحلة 1: أساسيات IAM (إدارة الهوية والصلاحيات)

هاي أول خطوة بأي حساب AWS حقيقي: **مين يقدر يعمل شو**.

### الخطوة 1.1: فهم Shared Responsibility Model

🔧 **شو بتعمل:** اقرأ/راجع مفهوم Shared Responsibility Model (بدون عملي).

📝 **شو سويت:** فهمت الحدود بين مسؤولية AWS ومسؤوليتك إنت.

⚙️ **وظيفته وميزاته:** هاد المفهوم بيتكرر بكل خدمة (EC2, Storage... الخ) وبيحدد مين مسؤول عن الأمان.

🧠 **أشياء لازم تعرفها:**
- **AWS مسؤولة عن أمان الـ Cloud** (البنية التحتية، الهاردوير، الشبكات الفيزيائية، الـ Data Centers).
- **إنت مسؤول عن الأمان *في* الـ Cloud** (IAM permissions, تشفير بياناتك, إعداد الـ Security Groups, تحديث نظام التشغيل جوا الـ EC2... الخ).
- هاد المفهوم بيجي بكل تقريبًا سؤال بالامتحان.

### الخطوة 1.2: إنشاء IAM Users و Groups

🔧 **شو بتعمل:**
1. من IAM Console → Users → أنشئ مستخدم جديد باسمك (بدل ما تستخدم Root بحياتك اليومية).
2. أنشئ Group اسمه مثلاً `developers` أو `admins`.
3. ألصق (Attach) عليه policy جاهزة زي `AdministratorAccess` (للتجربة بس — بمشروع حقيقي بتكون أدق من هيك).
4. ضيف المستخدم للـ Group.
5. سجّل خروج من Root وادخل بهاد المستخدم الجديد من الآن فصاعدًا.

📝 **شو سويت:** بدل ما أشتغل بحساب Root (اللي إله صلاحيات كاملة وخطير جدًا)، عملت مستخدم محدود الصلاحيات أشتغل فيه يوميًا، ونظّمت الصلاحيات عن طريق Group مش شخص-شخص.

⚙️ **وظيفته وميزاته:**
- **الأمان:** لو انسرق حساب المستخدم العادي، الضرر أقل بكثير من لو انسرق الـ Root.
- **Groups** بتخليك تدير صلاحيات عشرات المستخدمين بضغطة وحدة بدل ما تعدّل كل مستخدم لحاله.

🧠 **أشياء لازم تعرفها:**
- **Root account** ما بينحط إله أي Policy مباشرة، وما لازم تستخدمه أبدًا للشغل اليومي.
- **IAM هي خدمة Global** — مش مرتبطة بريجن معين.
- **مبدأ Least Privilege**: أعطي أقل صلاحية ممكنة تخلي الشغل يمشي، مش أكتر.
- Users ما بينضافوا لـ Group تلقائيًا — لازم تضيفهم يدويًا.

### الخطوة 1.3: كتابة IAM Policy مخصصة (Custom Policy)

🔧 **شو بتعمل:**
1. IAM → Policies → Create Policy.
2. اكتب Policy بصيغة JSON تسمح فقط بـ:
   - `ec2:Describe*` و `ec2:StartInstances` و `ec2:StopInstances` (مو Terminate!)
3. سمّيها `EC2LimitedAccess` وألصقها على Group تاني اسمه `interns` (تخيّل إنه فريق متدرب ما بدك يمسح أشياء بالغلط).

📝 **شو سويت:** كتبت صلاحية مخصصة تحدد بالضبط شو مسموح (Actions) وعلى أي مصادر (Resources)، بدل ما أعتمد على Policies جاهزة واسعة.

⚙️ **وظيفته وميزاته:** بالشغل الحقيقي نادرًا ما تستخدم `AdministratorAccess` — دايمًا في فريق (متدربين، فريق مراقبة، فريق دعم) لازم صلاحياته محدودة جدًا. هاد أساس مبدأ Least Privilege عمليًا.

🧠 **أشياء لازم تعرفها:**
- بنية الـ IAM Policy: `Effect` (Allow/Deny) + `Action` + `Resource` + (اختياري) `Condition`.
- **Explicit Deny دايمًا بيغلب أي Allow** حتى لو من Policy تانية.
- في 3 أنواع Policies: **Identity-based** (على User/Group/Role)، **Resource-based** (على المصدر نفسه زي S3 bucket policy)، و**Permissions boundaries**.

### الخطوة 1.4: تفعيل MFA وسياسة كلمات المرور (Password Policy)

🔧 **شو بتعمل:**
1. على المستخدم تبعك: Security credentials → فعّل **MFA** (فيرتشوال زي Google Authenticator).
2. IAM → Account settings → عدّل **Password Policy**: طول أدنى 8-12، لازم أرقام ورموز، Expiration بعد 90 يوم، منع إعادة استخدام آخر 5 كلمات مرور.

📝 **شو سويت:** أضفت طبقة حماية ثانية (MFA) لتسجيل الدخول، وفرضت قواعد قوية على كلمات المرور لكل المستخدمين.

⚙️ **وظيفته وميزاته:** حتى لو انسرقت كلمة السر، ما حدا يقدر يدخل بدون الرمز من جهازك (MFA). Password Policy بتمنع كلمات سر ضعيفة زي `123456`.

🧠 **أشياء لازم تعرفها:**
- MFA = **Something you know** (كلمة السر) + **Something you have** (جهازك/الرمز المؤقت).
- كل مستخدم بيقدر يدير كلمة سره، بس الـ Password Policy تُفرض على كل الحساب من مكان واحد.

### الخطوة 1.5: إنشاء Access Keys واستخدام AWS CLI

🔧 **شو بتعمل:**
1. أنشئ Access Key من Security Credentials (خزّنه بمكان آمن، ما بيظهر مرة ثانية).
2. نزّل وثبّت AWS CLI (Windows/Mac/Linux — حسب جهازك).
3. شغّل `aws configure` وحط الـ Access Key + Secret Key + Region + output format.
4. جرّب أمر بسيط زي `aws ec2 describe-regions` أو `aws s3 ls`.

📝 **شو سويت:** ربطت جهازي (Terminal) بحساب AWS تبعي، وصرت أقدر أنفذ أوامر AWS من سطر الأوامر بدل ما أدخل كل مرة عالـ Console.

⚙️ **وظيفته وميزاته:** أي أتمتة (Automation) أو Scripts أو DevOps شغل لازم يمر عبر الـ CLI أو الـ SDK — مش عبر المتصفح. هاد أول خطوة نحو أتمتة البنية التحتية.

🧠 **أشياء لازم تعرفها:**
- **Access Key ID + Secret Access Key** = زي username/password بس للبرمجة، ما تشاركهم أبدًا ولا ترفعهم على GitHub.
- في 3 طرق أساسية للوصول لـ AWS: **Management Console** (المتصفح)، **CLI**، و **SDK** (بالكود مباشرة، زي Python/JavaScript).
- لو سربت Access Key بالغلط، لازم تعطلها (Deactivate) فورًا من IAM.

### الخطوة 1.6: إنشاء IAM Role (بدون استخدام فعلي بعد)

🔧 **شو بتعمل
:** IAM → Roles → Create Role → اختار Trusted Entity = AWS Service → EC2 → ألصق Policy `AmazonS3ReadOnlyAccess` → سمّيه `EC2-S3-ReadOnly-Role`.

📝 **شو سويت:** جهزت "دور" (Role) بصلاحيات محددة، جاهز يُلصق لاحقًا على أي EC2 instance.

⚙️ **وظيفته وميزاته:** الـ Role هو الطريقة **الصحيحة والآمنة** لإعطاء EC2 instance صلاحيات — بدل ما تحط Access Keys جوا السيرفر (يلي هاد خطر أمني كبير جدًا لو انسرق السيرفر).

🧠 **أشياء لازم تعرفها:**
- الفرق بين **IAM User** (لشخص أو تطبيق دائم) و **IAM Role** (صلاحيات مؤقتة تُفترض Assumed من خدمة أو شخص).
- Roles
  Z بتستخدم **Temporary Security Credentials** بتتجدد تلقائيًا — ما في Access Key ثابت مخزّن بالسيرفر.
- Role ممكن تُلصق فقط وقت إنشاء الـ EC2 أو حتى بعدين على instance شغال.

### الخطوة 1.7: تقرير الـ Credentials Report و IAM Best Practices

🔧 **شو بتعمل:** IAM → Credential Report → Download وافحصه (بيوريك آخر استخدام لكل access key وكلمة سر ولو في MFA مفعّل).

📝 **شو سويت:** أخذت تقرير شامل عن حالة أمان كل المستخدمين بالحساب.

⚙️ **وظيفته وميزاته:** أداة تدقيق أمني (Security Audit) — بتساعدك تكتشف مستخدمين ما فعّلوا MFA أو Access Keys قديمة ما استُخدمت من مدة طويلة (لازم تُحذف).

🧠 **أشياء لازم تعرفها:**
- في أداة تانية بالامتحان اسمها **IAM Access Advisor** بتوريك آخر مرة كل صلاحية استُخدمت فعليًا (مفيدة لتقليل الصلاحيات الزايدة).
- Best Practices: ما تستخدم Root، فعّل MFA للكل، دوّر (Rotate) الـ Access Keys بشكل دوري، استخدم Roles بدل Keys جوا EC2.

## المرحلة 2: EC2 — تشغيل أول سيرفر

### الخطوة 2.1: إطلاق أول EC2 Instance

🔧 **شو بتعمل:**
1. EC2 Console → Launch Instance.
2. اختار AMI: **Amazon Linux 2**.
3. اختار Instance Type: **t2.micro** (ضمن الفري تير).
4. أنشئ Key Pair جديد (.pem) ونزّله واحفظه بمكان آمن.
5. Network Settings: أنشئ Security Group جديدة تسمح بـ:
   - SSH (Port 22)
      من IP تبعك بس (مو 0.0.0.0/0)
   - HTTP (Port 80) 
     من أي مكان (Anywhere)
1. Launch.

📝 **شو سويت:** شغّلت أول سيرفر فيرتشوال (Virtual Machine) على AWS، وحددت جدار حماية (Firewall) أساسي إله.

⚙️ **وظيفته وميزاته:** EC2 = **Elastic Compute Cloud**، هي الخدمة اللي بتعطيك سيرفرات (compute power) بالطلب بدون ما تشتري هاردوير فعلي. بتقدر تشغّل، توقف، تكبّر، تصغّر السيرفر بدقايق.

🧠 **أشياء لازم تعرفها:**
- **AMI** (Amazon Machine Image) = 
  "قالب" فيه نظام التشغيل + إعدادات مسبقة، منه بتنطلق أي instance.
- **Key Pair**: مفتاح خاص (Private Key بحاسوبك) + مفتاح عام (Public Key بالسيرفر) — يستخدموا للدخول الآمن عبر SSH بدل كلمة سر.
- **Security Group** = جدار حماية على مستوى الـ instance، **Stateful** (لو سمحت بطلب داخل، الرد يرجع تلقائيًا بدون قاعدة إضافية)، وكل قواعدها Allow بس (ما في Deny).
- t2.micro = ضمن Free Tier (750 ساعة بالشهر لأول سنة).

### الخطوة 2.2: الاتصال بالسيرفر — 3 طرق

🔧 **شو بتعمل:** جرّب الاتصال بنفس الـ instance بثلاث طرق مختلفة:
1. **SSH من Terminal** (Mac/Linux) أو PowerShell (Windows 10+): 
   `chmod 400 MyKey.pem` ثم `ssh -i MyKey.pem ec2-user@<Public-IP>`
2. **PuTTY** (لو Windows قديم): حوّل الـ .pem لـ .ppk عن طريق PuTTYgen، ثم اتصل.
3. **EC2 Instance Connect** (من المتصفح مباشرة، بدون أي مفتاح يدوي).

📝 **شو سويت:** جربت أكتر من طريقة أدخل فيها على نفس السيرفر عن بعد، وفهمت الفرق بينهم.

⚙️ **وظيفته وميزاته:** SSH هو الطريقة الأساسية للتحكم بأي سيرفر Linux عن بعد عن طريق سطر الأوامر. EC2 Instance Connect حل سريع وأبسط للتجربة والاختبار السريع من المتصفح.

🧠 **أشياء لازم تعرفها:**
- لازم تعمل `chmod 400` على الـ .pem file على Mac/Linux وإلا SSH برفض الاتصال (unprotected key file).
- المستخدم الافتراضي لـ Amazon Linux 2 هو `ec2-user`.
- الـ **Public IP بيتغيّر** كل مرة توقف وتشغّل السيرفر من جديد (إلا إذا استخدمت Elastic IP).
- لازم بورت 22 يكون مفتوح بالـ Security Group.

### الخطوة 2.3: تجربة User Data — أتمتة إعداد السيرفر

🔧 **شو بتعمل:** عند إطلاق instance جديد، بخانة **User Data** حط سكريبت Bash بسيط يثبّت Apache/Nginx ويعرض صفحة HTML بسيطة تلقائيًا وقت التشغيل الأول.

📝 **شو سويت:** خليت السيرفر يجهّز نفسه لحاله (تثبيت سيرفر ويب + عرض صفحة) بدون ما أدخل يدويًا وأكتب أوامر.

⚙️ **وظيفته وميزاته:** هاد أساس فكرة الـ **Auto Scaling** لاحقًا — أي instance جديد لازم يجهّز نفسه أوتوماتيكيًا بدون تدخل بشري، عشان تقدر تفتح عشرات السيرفرات بنفس اللحظة.

🧠 **أشياء لازم تعرفها:**
- User Data script بيشتغل مرة وحدة بس **أول تشغيل** للـ instance (Root user privileges تلقائيًا).
- هاد هو نفس السكريبت اللي رح تحتاجه لاحقًا بالـ Launch Template لما تعمل Auto Scaling Group.

### الخطوة 2.4: فهم Instance Types وPurchasing Options

🔧 **شو بتعمل:** (نظري + استكشاف) افتح صفحة اختيار Instance Type وقارن بين فئات مختلفة (t, m, c, r...)، وراجع خيارات الشراء بدون شراء فعلي.

📝 **شو سويت:** فهمت الفرق بين أنواع الأجهزة المتاحة وطرق الدفع المختلفة.

⚙️ **وظيفته وميزاته:** اختيار نوع الـ Instance الصح بيوفرلك مصاري كتير ويحسّن الأداء حسب نوع شغلك (معالجة، ذاكرة، شبكة).

🧠 **أشياء لازم تعرفها (مهم جدًا للامتحان):**
- فئات الـ Instance: **T** (general purpose, burstable)، **M** (general purpose)، **C** (Compute optimized)، **R** (Memory optimized)... الخ.
- **On-Demand**: تدفع بالساعة/الثانية بدون التزام — الأغلى بس الأمرن.
- **Reserved Instances**: التزام 1-3 سنوات، خصم كبير، للأحمال الثابتة المستمرة.
- **Savings Plans**: التزام بمبلغ إنفاق ثابت بالساعة، مرن أكتر من Reserved.
- **Spot Instances**: الأرخص بكثير (خصم يوصل 90%) بس ممكن AWS توقفك بأي وقت (لأعمال مش حساسة للانقطاع).
- **Dedicated Hosts/Instances**: سيرفر فيزيائي مخصص لك بالكامل (لمتطلبات Compliance/Licensing).

### الخطوة 2.5: فهم رسوم IPv4 الجديدة (Public IPv4 Charges)

🔧 **شو بتعمل:** راجع Elastic IPs الموجودة عندك، تأكد ما في وحدة "معلّقة" (Unassociated) مربوطة بحسابك بدون استخدام.

📝 **شو سويت:** تأكدت إني مش بادفع على IP عام مش مستخدم فعليًا.

⚙️ **وظيفته وميزاته:** من فبراير 2024، AWS صارت تحاسب على أي Public IPv4 (حتى لو مش Elastic IP) — لازم تنتبه وتحرر أي IP مش مستخدم.

🧠 **أشياء لازم تعرفها:**
- Elastic IP = عنوان IP عام ثابت ما بيتغيّر حتى لو أوقفت/شغّلت الـ instance — لكن **لو ما ربطته بأي instance، بتُحاسب عليه إضافي** كتحفيز إنك تحرره.

## المرحلة 3: التخزين (Storage) و AMI

### الخطوة 3.1: استكشاف EBS Volume المرتبط بالـ Instance

🔧 **شو بتعمل:** من EC2 Console → Instances → اختار الـ instance تبعك → تبويب Storage → شوف الـ EBS volume المرتبط فيه (Root Volume).

📝 **شو سويت:** شفت إن السيرفر تبعي فعليًا بيخزن بياناته على "قرص شبكي" منفصل (Network Drive) مش قرص فيزيائي داخل نفس الجهاز.

⚙️ **وظيفته وميزاته:** **EBS (Elastic Block Store)** هو قرص تخزين شبكي يبقى موجود حتى لو حذفت الـ instance (إذا اخترت هيك)، ويقدر ينتقل بين instances بنفس الـ Availability Zone.

🧠 **أشياء لازم تعرفها:**
- EBS مربوط بـ **Availability Zone واحدة بس** — ما تقدر تربطه مباشرة بـ instance بمنطقة (AZ) تانية.
- فيه أنواع: **gp2/gp3** (SSD عام)، **io1/io2** (SSD عالي الأداء للـ Databases)، **st1/sc1** (HDD رخيص للأرشفة).
- Volume بيتفصل عن الـ Root إذا حددت "Delete on Termination = false"، وبيبقى موجود حتى لو انحذف الـ instance.

### الخطوة 3.2: أخذ EBS Snapshot (نسخة احتياطية)

🔧 **شو بتعمل:** اختار الـ EBS Volume → Actions → Create Snapshot، وسمّيه.

📝 **شو سويت:** أخذت نسخة احتياطية (Backup) للحالة الحالية للقرص كامل.

⚙️ **وظيفته وميزاته:** لو صار أي عطل أو حذف بالغلط، تقدر تسترجع البيانات بإنشاء Volume جديد من الـ Snapshot، أو تنقل بياناتك بين Regions/AZs.

🧠 **أشياء لازم تعرفها:**
- Snapshots 
  تُخزّن على **S3** بالخلفية (مع إنك ما بتتعامل معها مباشرة كـ S3 objects).
- تقدر تعمل Snapshot وأنت الـ Volume شغال (Live)، لكن الأفضل تعمله وقت التوقف لضمان تناسق البيانات (Consistency).
- من الـ Snapshot تقدر تنشئ Volume بأي Availability Zone أو حتى تنسخه (Copy) لـ Region تاني بالكامل.

### الخطوة 3.3: إنشاء AMI مخصص من الـ Instance

🔧 **شو بتعمل:** من الـ instance الشغال (اللي ثبتنا عليه سيرفر الويب بالخطوة 2.3) → Actions → Image and templates → Create Image.

📝 **شو سويت:** حوّلت السيرفر المُجهّز بالكامل (مع الويب سيرفر المثبت والإعدادات) إلى "قالب" (Template) جديد خاص فيني.

⚙️ **وظيفته وميزاته:** بدل ما تعيد تثبيت وإعداد كل شي من الصفر كل مرة، أي instance جديد تطلقه من هاد الـ AMI رح يجي جاهز 100% من أول ثانية — **هاد أساس Auto Scaling لاحقًا**.

🧠 **أشياء لازم تعرفها:**
- AMI مرتبط بـ **Region واحد** بس (لازم تنسخه Copy لو بدك تستخدمه بمنطقة تانية).
- AMI = صورة كاملة لنظام التشغيل + البرامج المثبتة + الإعدادات، مش بس Storage.
- الفرق بين **Instance Store** (قرص فيزيائي ملتصق بالهاردوير مباشرة، أسرع بكثير بس **يضيع البيانات** إذا وقف/انطفى الـ instance — Ephemeral Storage) و **EBS** (شبكي، دائم/Persistent).
- خدمة **EC2 Image Builder** (لو بدك تتوسع): بتؤتمت إنشاء واختبار وتوزيع الـ AMIs بشكل دوري ومجدول.

### الخطوة 3.4 (اختياري توسّعي): فهم EFS

🔧 **شو بتعمل:** (نظري) راجع الفرق بين EBS و EFS من خلال جدول مقارنة تسويه بنفسك.

📝 **شو سويت:** فهمت متى أستخدم EFS بدل EBS.

⚙️ **وظيفته وميزاته:** **EFS (Elastic File System)** هو نظام ملفات (NFS) يقدر يترَبط بأكثر من instance بنفس الوقت وعبر أكثر من Availability Zone — بعكس EBS اللي مرتبط بـ instance واحد وAZ واحدة.

🧠 **أشياء لازم تعرفها:**
- EFS مثالي لما يكون عندك أكثر من سيرفر (زي بالـ Auto Scaling Group بالمرحلة الجاية) وكلهم لازم يوصلوا لنفس الملفات المشتركة بنفس الوقت.
- EFS أغلى من EBS لكل GB، بس بيدعم مشاركة فعلية بين أجهزة متعددة.

### الخطوة 3.5: التنظيف الجزئي

🔧 **شو بتعمل:** احذف أي Snapshot أو AMI تجريبي زايد ما رح تحتاجه (خصوصًا لو خارج الفري تير).

📝 **شو سويت:** نظّفت الموارد غير المستخدمة لتفادي تكاليف زائدة.

⚙️ **وظيفته وميزاته:** EBS Snapshots و AMIs (والـ EBS volumes خلفها) بتكلّف مصاري تخزين حتى لو مش مستخدمة فعليًا.

🧠 **أشياء لازم تعرفها:** لازم تحذف الـ Snapshot **و**كمان الـ backing EBS snapshot المرتبط بالـ AMI (deregister AMI لحاله ما يكفي، لازم تحذف الـ snapshot كمان).

## المرحلة 4: High Availability و Scalability — ELB & ASG

هاي أهم مرحلة وبتربط كل شي سويته قبل هلق مع بعض.

### الخطوة 4.1: تجهيز 2 (أو أكتر) EC2 Instances متطابقين

🔧 **شو بتعمل:** أطلق instance ثاني من نفس الـ **AMI المخصص** اللي عملته بالخطوة 3.3 (بحيث يجي جاهز فيه سيرفر الويب من غير أي إعداد يدوي إضافي).

📝 **شو سويت:** صار عندي أكثر من سيرفر متطابق، جاهزين يستقبلوا Traffic بنفس الوقت.

⚙️ **وظيفته وميزاته:** أساس الـ **High Availability**: لو وقع سيرفر واحد، السيرفر التاني لسا شغال والموقع ما بيوقف.

🧠 **أشياء لازم تعرفها:** لازم الـ instances تكون بأكثر من **Availability Zone** مختلفة (مش بس أكثر من instance بنفس الـ AZ) عشان لو وقعت منطقة كاملة (نادر بس ممكن)، يضل عندك سيرفر شغال بمنطقة تانية.

### الخطوة 4.2: إنشاء Application Load Balancer (ALB)

🔧 **شو بتعمل:**
1. EC2 Console → Load Balancers → Create → **Application Load Balancer**.
2. اختار كل الـ Availability Zones المتاحة.
3. أنشئ **Target Group** وضيف فيه الـ instances الاثنين (على بورت 80).
4. أنشئ Security Group خاصة بالـ Load Balancer تسمح HTTP من الكل (Anywhere).
5. بعد التفعيل، افتح الـ DNS Name تبع الـ Load Balancer بالمتصفح.

📝 **شو سويت:** حطيت "بوابة" واحدة (DNS واحد) قدام السيرفرين، وهي يلي رح توزع الطلبات بينهم تلقائيًا.

⚙️ **وظيفته وميزاته:**
- المستخدم ما بيتعامل مباشرة مع أي IP لسيرفر معين — بس مع الـ Load Balancer، يلي بدوره بيوزع الحمل (Traffic).
- لو وقع سيرفر واحد، الـ Load Balancer بيوقف يبعتله طلبات تلقائيًا (Health Checks) ويكمل ع الباقي.

🧠 **أشياء لازم تعرفها:**
- 3 أنواع Load Balancers: **ALB** (Application — طبقة 7، للـ HTTP/HTTPS، بيدعم Routing بناءً على المسار/الدومين)، **NLB** (Network — طبقة 4، للأداء العالي جدًا وTCP)، **GWLB** (Gateway — لأدوات الأمان الطرفية Third-party).
- **Health Checks**: الـ Load Balancer بيفحص كل instance بشكل دوري (مثلًا HTTP GET على `/`) وإذا فشل، بيوقف التوجيه له تلقائيًا لحد ما يرجع "Healthy".
- Load Balancer عنده **DNS Name ثابت**، مش IP ثابت (لأنه ممكن يتوسع بالخلفية لأكثر من عقدة).
- Security Group تبع الـ instances لازم تسمح بالترافيك **من الـ Security Group تبع الـ Load Balancer نفسها**، مش من الكل مباشرة (أفضل ممارسة أمنية).

### الخطوة 4.3: إنشاء Launch Template

🔧 **شو بتعمل:** EC2 → Launch Templates → Create، واستخدم فيه الـ AMI المخصص + Instance type + Security Group + Key Pair + نفس User Data (لو حابب).

📝 **شو سويت:** جهزت "وصفة" كاملة يقدر AWS يستخدمها لإطلاق instances جديدة تلقائيًا بدون أي تدخل مني.

⚙️ **وظيفته وميزاته:** هاد أساس الـ Auto Scaling — بدون Template، AWS ما بيعرف "كيف" يطلق instance جديد تلقائيًا.

🧠 **أشياء لازم تعرفها:** Launch Template هو النسخة المحدّثة (وتقريبًا الوحيدة المستخدمة اليوم) بدل الـ Launch Configuration القديمة، ويدعم عدة نُسخ (Versions).

### الخطوة 4.4: إنشاء Auto Scaling Group (ASG)

🔧 **شو بتعمل:**
1. EC2 → Auto Scaling Groups → Create → اختار الـ Launch Template.
2. اربطه بالـ Target Group تبع الـ Load Balancer (نفس المرحلة 4.2).
3. حدد: Desired = 2، Minimum = 2، Maximum = 4.
4. اختار الـ Availability Zones المتعددة.

📝 **شو سويت:** ربطت كل القطع مع بعض: Launch Template + Load Balancer + عدد Instances المطلوب، وخليت AWS يدير عدد السيرفرات تلقائيًا.

⚙️ **وظيفته وميزاته:** لو وقع instance بالغلط أو انحذف، الـ ASG بيلاحظ إن العدد الحالي أقل من الـ Desired ويطلق واحد جديد تلقائيًا بديل عنه — بدون أي تدخل بشري.

🧠 **أشياء لازم تعرفها:**
- **Min / Desired / Max** هي الإعدادات الأساسية الثلاثة لأي ASG.
- الـ ASG بيتكامل مباشرة مع الـ Target Group تبع الـ Load Balancer — أي instance جديد بينضم أوتوماتيكيًا للتوزيع.
- ASG **مجاني** — بتدفع بس على الـ EC2 instances نفسها اللي بيطلقها.

### الخطوة 4.5: إعداد Scaling Policies (استراتيجيات التوسع)

🔧 **شو بتعمل:** بالـ ASG → Automatic Scaling → أضف Policy من نوع **Target Tracking** تستهدف مثلًا 40% متوسط استخدام CPU.

📝 **شو سويت:** خليت AWS يزيد أو يقلل عدد السيرفرات تلقائيًا حسب الحمل الفعلي (Metric محدد)، مش عدد ثابت طول الوقت.

⚙️ **وظيفته وميزاته:** لو زاد الضغط ع الموقع (مثلًا CPU طلع فوق 40%)، الـ ASG بيطلق instance جديد تلقائيًا لغاية ما يرجع الاستخدام للمعدل المستهدف — وبالعكس، لو قل الضغط بيقفل instances زيادة (بس مش تحت الـ Minimum).

🧠 **أشياء لازم تعرفها (استراتيجيات Scaling الرئيسية):**
- **Target Tracking Scaling**: الأسهل — بس تحدد Metric مستهدف (زي CPU 40%) وAWS يتصرف لحاله.
- **Simple/Step Scaling**: تحدد يدويًا CloudWatch Alarms (زي "لو CPU > 70% لمدة 5 دقائق زيد instance") — تحكم أدق بس أعقد.
- **Scheduled Scaling**: تعرف مسبقًا أوقات ضغط متوقعة (مثلًا كل صباح جمعة) وتزيد العدد مسبقًا قبل ما يوصل الضغط فعليًا.
- **Predictive Scaling**: يستخدم Machine Learning يتوقع الحمل المستقبلي بناءً على أنماط سابقة.
- **Cooldown Period**: فترة انتظار بعد أي عملية Scaling قبل ما يسمح بعملية تانية، عشان يمنع "تذبذب" مستمر (Scaling Flapping).

### الخطوة 4.6: اختبار كامل المنظومة

🔧 **شو بتعمل:**
1. افتح الـ Load Balancer DNS بالمتصفح كذا مرة وشوف إنه بيتنقل بين السيرفرين.
2. جرب تنهي (Terminate) instance واحد يدويًا وراقب كيف الـ ASG بيطلق واحد بديل تلقائيًا خلال دقايق.
3. (اختياري) استخدم أداة ضغط بسيطة على أحد السيرفرات وشوف كيف الـ CPU بيطلع والـ ASG بيتفاعل.

📝 **شو سويت:** أثبتّ عمليًا إن النظام يلي بنيته فعلاً **Highly Available** (بيصمد لو وقع جزء منه) و**Scalable** (بيتوسع ويتقلص لحاله حسب الحاجة).

⚙️ **وظيفته وميزاته:** هاد بالضبط جوهر الـ Cloud Computing — أنظمة بتصلح نفسها وتتوسع لحالها، بدون ما حدا يصحى الساعة 3 الفجر يطلق سيرفر يدويًا.

🧠 **أشياء لازم تعرفها:**
- **High Availability** ≠ **Fault Tolerance**: HA يعني الخدمة بتضل شغالة (بأداء أقل ممكن)، أما Fault Tolerance يعني ما في أي انقطاع حتى لحظي.
- **Scalability** له نوعين: **Vertical** (تكبير نفس السيرفر — حجم واحد بس أقوى) و **Horizontal** (زيادة عدد السيرفرات — هاد يلي عملناه بالـ ASG).

## المرحلة 5: التنظيف النهائي (Cleanup)

🔧 **شو بتعمل (بالترتيب الصحيح، مهم جدًا):**
1. احذف الـ Auto Scaling Group أولًا (رح يحذف الـ instances تبعه تلقائيًا).
2. احذف الـ Load Balancer.
3. احذف أي Target Groups يتيمة.
4. احذف أي EC2 instances يدوية باقية.
5. احذف الـ EBS Volumes/Snapshots و AMIs التجريبية.
6. احذف الـ Elastic IPs غير المربوطة (لو في).
7. راجع Billing Dashboard تأكد ما في شي شغال بالخلفية.

📝 **شو سويت:** نظفت كل الموارد بترتيب يحترم الاعتماديات بينها (Dependencies) عشان ما تعلق حذف شي بسبب شي تاني مرتبط فيه.

⚙️ **وظيفته وميزاته:** أهم عادة لازم تتعلمها بـ AWS — نسيان مورد شغال هو السبب الأكثر شيوعًا لفواتير مفاجئة.

🧠 **أشياء لازم تعرفها:** ترتيب الحذف مهم — مثلًا ما تقدر تحذف Target Group وهو لسا مربوط بـ Load Balancer، وما تقدر تحذف EBS Volume وهو Root Volume لـ instance شغال.

## خلاصة: خريطة المفاهيم يلي طبقتها

| الخدمة | دورها بالمشروع |
|---|---|
| **IAM** | إدارة مين يقدر يوصل لأي مورد وبأي صلاحية (Users, Groups, Policies, Roles, MFA) |
| **EC2** | السيرفرات الفعلية اللي بتشغل التطبيق |
| **Security Groups** | جدار الحماية على مستوى كل instance |
| **EBS** | التخزين الدائم المرتبط بكل instance |
| **AMI** | القالب الجاهز لإطلاق instances متطابقة بسرعة |
| **Launch Template** | "الوصفة" اللي بتستخدمها الـ ASG لإطلاق instances جديدة |
| **ALB (Load Balancer)** | نقطة دخول واحدة، بتوزع الترافيك وبتراقب صحة السيرفرات |
| **Auto Scaling Group** | بيحافظ على عدد السيرفرات الصح وبيتوسع/يتقلص تلقائيًا |
| **Budgets/Billing** | مراقبة التكلفة طول الوقت |
