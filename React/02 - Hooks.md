### 1. الخلاصة العامة (High-Level Summary)

يُمهد هذا الفيديو لمفهوم الـ **Hooks** في React ويوضح الأسباب الجوهرية التي تمنعنا من استخدام المتغيرات العادية لإدارة حالة التطبيق داخل المكونات. كما يستعرض الشارح بناء مكون تفاعلي لإنشاء طلبات (Order Form)، مع التركيز على أهمية كتابة كود نقي وسريع لتجنب مشاكل الأداء، ويقدم نصيحة معمارية حول كيفية تعريف المكونات لتسهيل تتبع الأخطاء لاحقاً.

### 2. المفاهيم والركائز الأساسية (Core Concepts)

#### **أ. مسار التصيير والأداء (Render Path & Performance)**

- **ما هو؟** هو الكود الموجود داخل جسم الـ Component والذي يحدد ما سيتم رسمه على الشاشة. وصفه الشارح بأنه "Hot Code Path"، أي أنه كود يتم تشغيله مراراً وتكراراً وبكثافة.
    
- **لماذا نستخدمه بذكاء؟ (المشكلة والحل):** إذا وضعت عمليات برمجية ثقيلة (Heavy Lifting) داخل هذا المسار، سيؤدي ذلك إلى بطء في عرض عناصر الـ DOM، مما ينتج عنه تقطيع في واجهة المستخدم (Jank). الحل هو إبقاء هذا الكود سريعاً جداً وخالياً من العمليات المعقدة.
    

#### **ب. التأثيرات الجانبية وإدارة البيانات (Side Effects & Unpredictability)**

- **ما هو؟** كتابة كود يغير متغيرات خارجية أو يعتمد على بيانات متغيرة ديناميكياً (مثل استخدام `Date.now()` أو عدّاد `counter++`) داخل جسم الـ Component مباشرة.
    
- **لماذا نتجنبه؟ (المشكلة والحل):** React هي التي تقرر متى تقوم بعملية الـ Re-render، وأنت كمطور لا تملك تحكماً مطلقاً في توقيت ذلك. إذا كان المكون الخاص بك يعتمد على تأثيرات جانبية غير مضبوطة، ستظهر نتائج عشوائية ومشاكل في الواجهة. الحل هو أن يكون المكون مبنياً بحيث يعطي دائماً نفس النتيجة (Markup) طالما أُعطي نفس المدخلات. إدارة هذه التأثيرات والحالة (State) ستتم لاحقاً بطرق آمنة عبر الـ Hooks (مثل `useEffect` و `useState`).
    

#### **ج. الدوال المسماة مقابل الدوال المجهولة (Named vs. Anonymous Functions)**

- **ما هو؟** طريقة تعريفك للـ Component في React. يمكنك استخدامه كدالة مسماة (Named Function) أو دالة سهمية مجهولة (Anonymous Arrow Function).
    
- **لماذا نستخدم الأولى؟ (المشكلة والحل):** عند حدوث انهيار في التطبيق (Crash)، تظهر الدوال السهمية المجهولة في سجل الأخطاء (Stack Trace) كدوال غير معروفة الاسم، مما يصعب عملية تتبع الخطأ (Debugging). أما الدوال المسماة، فتظهر باسمها الصريح، مما يوفر عليك وقتاً كبيراً في البحث عن مصدر المشكلة.
    
- **كيف يعمل؟ (مثال برمجي):**
    



```JavaScript
// الطريقة الأقل تفضيلاً (ستظهر كمكون مجهول في سجل الأخطاء)
const Order = () => {
  return <div>Order Page</div>;
};
export default Order;

// الطريقة الأفضل والموصى بها لتسهيل الـ Debugging
export default function Order() {
  return <div>Order Page</div>;
}
```

### 3. الخطوات العملية (Step-by-Step Guide)

قام الشارح خلال الفيديو بالبدء في بناء صفحة الطلبات من الصفر. إليك الخطوات العملية المتبعة:

1. **إنشاء الملف:** إنشاء ملف جديد باسم `Order.jsx`.
    
2. **تعريف المكون:** كتابة المكون كـ Named Function وتصديره (`export default function Order`).
    
3. **تجهيز البيانات الوهمية:** تعريف متغيرات ثابتة مؤقتاً لتمثيل حالة الطلب (State)، مثل: `const pizzaType = "pepperoni";` و `const pizzaSize = "M";`.
    
4. **بناء نموذج الطلب (Form):**
    
    - إضافة عنوان للصفحة.
        
    - إنشاء قائمة منسدلة `<select>` لاختيار نوع البيتزا.
        
    - إنشاء أزرار اختيار `<input type="radio">` لاختيار الحجم (S, M, L)، مع ربط حالة الـ `checked` بالمتغير المؤقت `pizzaSize`.
        
    - إضافة زر الإرسال (Submit Button) لإضافة الطلب إلى السلة.
        
5. **تضمين المكون الفرعي:** إضافة مكون البيتزا داخل نموذج الطلب كمعاينة بصرية للمستخدم (مع تمرير بعض البيانات الثابتة حالياً مثل الصورة والسعر).
    
6. **تحديث التطبيق الرئيسي:** الانتقال إلى ملف `App.jsx`، إزالة مكونات البيتزا المستوردة مسبقاً، استيراد مكون `Order` الجديد، وجعله هو الصفحة الرئيسية المعروضة.
    

### 4. الأخطاء الشائعة أو النصائح (Common Pitfalls & Tips)

- **تحذير من الأداء (Jank):** إياك ووضع عمليات معالجة بيانات ضخمة داخل دالة الـ Render. واجهة المستخدم البطيئة تجعل المستخدم يشعر بأن التطبيق "مكسور" أو غير احترافي.
    
- **تحذير من المتغيرات الخارجية:** محاولة تتبع حالة التطبيق (Statefulness) باستخدام متغيرات جافا سكريبت العادية (مثل `let counter = 0`) خارج أو داخل المكون مباشرة هي **فكرة سيئة جداً** في React؛ لأن التحديث عليها لن يخبر React بإعادة رسم الواجهة، والـ Re-render العشوائي سيقوم بتدمير أو تكرار هذه القيم بشكل غير متوقع.
    
- **نصيحة هيكلية:** رغم أن استخدام الـ Arrow Functions يبدو أنيقاً ومختصراً، إلا أن التضحية بها لصالح الـ Named Functions في تعريف الـ Components الأساسية هو خيار أفضل للمشاريع الكبيرة بسبب سهولة قراءة الـ Stack Trace.
    

### 5. قاموس المصطلحات السريع

- **Hooks:** 
  دوال مدمجة في React تتيح لك إضافة "حالة" (State) وتفاعلية (Side Effects) للمكونات الدالية (Functional Components).
    
- **Render Path:** 
  المسار أو الكود الذي يُنفذ لبناء وإرجاع عناصر واجهة المستخدم (DOM) داخل المكون.
    
- **Re-render:**
   دورة يقوم فيها React بإعادة قراءة كود المكون وتحديث الشاشة إذا اكتشف وجود تغييرات في البيانات.
    
- **Jank:** 
  مصطلح يصف التقطيع، الانخفاض في سرعة الإطارات، أو الاستجابة البطيئة للمتصفح نتيجة
   كود غير مُحسّن.
    
- **Stack Trace:**
   تقرير مفصل يقرؤه المبرمجون في شاشة الـ Console، يتتبع تسلسل الدوال التي تم استدعاؤها لحظة حدوث خطأ برمجي للوصول إلى جذره.
    
- **Side Effect (تأثير جانبي):** 
  أي تفاعل يقوم به المكون مع العالم الخارجي ولا يقتصر فقط على
   تحويل الـ Props إلى واجهة مستخدم (مثل استدعاء واجهة برمجة تطبيقات API، أو التعامل مع الزمن كـ `Date.now()`).

إليك التحليل والملخص الشامل للجزء الجديد من الفيديو، استكمالاً للمنهجية السابقة:

### 1. الخلاصة العامة (High-Level Summary)

يُركز هذا المقطع على توضيح سبب فشل النماذج (Forms) في React عند الاعتماد على متغيرات عادية، ويُقدم مفهوم الـ **Hooks** وتحديداً **`useState`** كحل جذري لإدارة الحالة (State). الهدف هو فهم كيفية تحويل المكونات الجامدة إلى مكونات تفاعلية مع احترام مبدأ التدفق أحادي الاتجاه للبيانات (One-way Data Flow) وقواعد كتابة الـ Hooks.

### 2. المفاهيم والركائز الأساسية (Core Concepts)

#### **أ. غياب الربط ثنائي الاتجاه التلقائي (No Free Two-way Data Binding)**

- **ما هو؟** في بعض إطارات العمل (مثل Angular القديم)، عندما يتغير المتغير تتغير الواجهة، وعندما يكتب المستخدم في الواجهة يتغير المتغير تلقائياً (ربط ثنائي). في React، هذا غير موجود؛ البيانات تتدفق في اتجاه واحد فقط.
    
- **لماذا نستخدم أسلوب React؟ (المشكلة والحل):** الربط الثنائي التلقائي يبدو ممتعاً وسريعاً في الكتابة، لكنه يتحول إلى "سحر أسود" (Black Magic) في المشاريع الكبيرة ويصعب تتبع مصدر الأخطاء. أسلوب React أكثر وضوحاً (Verbose)، مما يجعل صيانة الكود أسهل بكثير ويوفر مئات الساعات من الـ Debugging لاحقاً.
    
- **كيف يعمل؟** لتحديث الواجهة بناءً على إدخال المستخدم، يجب أن نلتقط الحدث يدوياً، ونحدث الحالة (State)، ثم تقوم React بإعادة رسم (Re-render) المكون بالقيمة الجديدة.
    

#### **ب. خطاف الحالة (useState Hook)**

- **ما هو؟** أي دالة في React تبدأ بكلمة `use` تُسمى Hook. أداة `useState` هي الأهم والأكثر استخداماً، وتسمح للمكون بحفظ بيانات تتغير بمرور الوقت.
    
- **لماذا نستخدمه؟ (المشكلة والحل):** المتغيرات العادية لا تُخبر React بضرورة إعادة تحديث واجهة المستخدم عند تغير قيمتها. الـ `useState` يحل هذه المشكلة؛ فعند تحديث قيمته، يتم عمل Re-render تلقائياً.
    
- **كيف يعمل؟ (مثال برمجي):** يستقبل الـ Hook قيمة ابتدائية، ويُرجع مصفوفة (Array) تحتوي على عنصرين: المتغير الذي يحمل القيمة الحالية، ودالة وظيفتها الوحيدة هي تحديث هذا المتغير.
    



```JavaScript
// استخدام Array Destructuring لاستخراج القيمة ودالة التحديث
const [pizzaType, setPizzaType] = useState("pepperoni");

// e.target.value تجلب القيمة التي اختارها المستخدم
<select onChange={(e) => setPizzaType(e.target.value)}>
```

#### **ج. قواعد الـ Hooks الصارمة (Rules of Hooks)**

- **ما هو؟** قيود معمارية تفرضها React على مكان وكيفية استدعاء أي Hook.
    
- **لماذا نستخدمه؟** تعتمد React على "الترتيب الدقيق" لاستدعاء الـ Hooks لتعرف أي حالة (State) تنتمي لأي متغير أثناء دورة التحديث (Re-render).
    
- **كيف يعمل؟** يُمنع وضع الـ Hooks داخل الشروط (`if`) أو حلقات التكرار (`for/while`). يجب أن يتم استدعاؤها دائماً في المستوى الأعلى (Top-level) للمكون ليظل ترتيب استدعائها ثابتاً في كل مرة.
    

### 3. الخطوات العملية (Step-by-Step Guide)

لمعالجة النموذج المكسور (الذي لا يقبل تغيير الخيارات)، قام الشارح بالخطوات التالية:

1. **استيراد الأداة:** إضافة `import { useState } from 'react'` في بداية الملف.
    
2. **استبدال المتغيرات العادية:**
    
    - تحويل `const pizzaType` إلى حالة: `const [pizzaType, setPizzaType] = useState('pepperoni');`
        
    - تحويل حجم البيتزا إلى حالة: `const [pizzaSize, setPizzaSize] = useState('M');`
        
3. **مراقبة التغييرات (Event Listeners):**
    
    - في عنصر القائمة المنسدلة `<select>`، تمت إضافة `onChange` مع دالة تلتقط الحدث `e` وتمرر القيمة الجديدة `e.target.value` إلى دالة التحديث `setPizzaType`.
        
    - تكرار نفس العملية لأزرار الراديو `<input type="radio">`، حيث تمت إضافة حدث `onChange` لكل زر على حدة لاستدعاء `setPizzaSize`.
        
4. **النتيجة:** بمجرد اختيار قيمة جديدة، تتحدث الـ State، وتقوم React بإعادة بناء المكون واختيار القيمة الصحيحة في المتصفح.
    

### 4. الأخطاء الشائعة أو النصائح (Common Pitfalls & Tips)

- **الفخ القاتل (Hooks inside Conditionals):** كتابة كود مثل `if (condition) { useState() }` سيكسر التطبيق بالكامل. تأكد دائماً أن الـ Hooks تعمل في كل عملية Re-render دون استثناء.
    
- **احذر من الاعتماد على الـ Event Bubbling في النماذج:** يمكنك تقنياً وضع حدث `onChange` على الـ `<div/>` الأب الذي يحوي أزرار الراديو، وسيعمل الكود بفضل ميزة انتقال الأحداث للأعلى (Bubbling). **ولكن**، هذا يعتبر ممارسة سيئة برمجياً (Anti-pattern) وستعترض عليه أدوات الـ Linters ومبادئ الـ Accessibility، لأن قارئات الشاشة للمكفوفين تتفاعل مع الـ `input` وليس الـ `div`. لذلك، ضع الحدث على الـ `<input>` مباشرة.
    
- **نصيحة الصيانة:** "لا تقم بتحسين الكود من أجل الساعتين التي ستكتب فيها المشروع، بل حسّن من أجل مئات الساعات التي ستقضيها في صيانته". (مبرر الشارح لكون React تتطلب كتابة كود أطول من غيرها).
    

### 5. قاموس المصطلحات السريع

- **Hook:** دوال خاصة مدمجة في React (تبدأ دائماً بكلمة use) تتيح لك استخدام الـ State وميزات React الأخرى داخل الـ Functional Components.
    
- **useState:** الـ Hook الأساسي المسؤول عن إعطاء المكون ذاكرة (State) للاحتفاظ بالمتغيرات وتحديث واجهة المستخدم عند تغيرها.
    
- **Array Destructuring:** ميزة في JavaScript تتيح استخراج عناصر من مصفوفة وتعيينها لمتغيرات في سطر واحد (مثل التوزيع المستخدم مع useState).
    
- **Two-way Data Binding:** نمط برمجي (غير مستخدم في React) يقوم تلقائياً بمزامنة قيمة المتغير البرمجي مع ما يعرض في واجهة المستخدم والعكس.
    
- **Event Bubbling:** خاصية في المتصفحات حيث "يفور" أو ينتقل الحدث (مثل النقر أو التغيير) من العنصر الداخلي إلى العناصر الأب التي تحتويه.
    
- **e.target.value:** خاصية داخل كائن الحدث (Event) في المتصفح، تُرجع القيمة الفعلية الموجودة داخل حقل الإدخال أو القائمة المنسدلة لحظة تفاعل المستخدم معها.

### 1. الخلاصة العامة (High-Level Summary)

يُركز هذا المقطع على ثاني أهم خطاف في React وهو **`useEffect`**، والذي يُستخدم للتعامل مع "التأثيرات الجانبية" (Side Effects) مثل جلب البيانات من الخوادم (APIs). كما يشرح الشارح كيفية ربط هذه البيانات المستلمة بواجهة المستخدم بشكل ديناميكي باستخدام دالة `.map()`، بالإضافة إلى مفهوم "الحالة المشتقة" (Derived State) لتقليل الاعتماد غير المبرر على الـ Hooks.

### 2. المفاهيم والركائز الأساسية (Core Concepts)

#### **أ. التأثيرات الجانبية وخطاف (useEffect Hook)**

- **ما هو؟** هو أداة تسمح لك بتنفيذ أوامر تتواصل مع العالم الخارجي (خارج إطار المكون نفسه)، مثل جلب بيانات من واجهة برمجة تطبيقات (API)، أو إعداد مؤقت (Timer).
    
- **لماذا نستخدمه؟ (المشكلة والحل):** إذا قمت بوضع كود جلب البيانات (Fetch) مباشرة داخل جسم الـ Component، سيتم استدعاء الخادم في كل مرة يحدث فيها Re-render، مما قد يؤدي إلى إغراق الخادم بالطلبات (بمثابة هجوم DDoS على خادمك). الحل هو استخدام `useEffect` لفصل هذه العمليات عن مسار التصيير (Render Cycle) والتحكم في توقيت تشغيلها.
    
- **كيف يعمل؟ (مثال برمجي):** يستقبل الـ Hook دالة (Function) ومصفوفة اعتماديات (Dependency Array).
    



```JavaScript
useEffect(() => {
  // الكود الذي سيتواصل مع الـ API
  fetchPizzaTypes();
}, []); // مصفوفة فارغة تعني: نفذ هذا الكود مرة واحدة فقط عند تحميل المكون
```

#### **ب. مصفوفة الاعتماديات (Dependency Array)**

- **ما هو؟** هو المعامل الثاني الذي يتم تمريره لدالة `useEffect` (العبارة عن الأقواس المربعة `[]`).
    
- **لماذا نستخدمه؟** لإخبار React بالمتغيرات التي يجب أن يراقبها. إذا تغيرت قيمة أحد المتغيرات داخل هذه المصفوفة، سيتم إعادة تشغيل الـ Effect. وإذا تُركت المصفوفة فارغة `[]`، سيعمل الـ Effect مرة واحدة فقط عند البداية (Mounting).
    

#### **ج. الحالة المشتقة (Derived State)**

- **ما هو؟** هي قيم برمجية تعتمد بالكامل في حسابها على قيم (State) أخرى موجودة مسبقاً.
    
- **لماذا نستخدمه؟ (المشكلة والحل):** بدلاً من إنشاء `useState` جديد لكل متغير (مثل السعر أو تفاصيل البيتزا المختارة)، يمكنك استنتاجها برمجياً من الـ State الأساسية باستخدام متغيرات جافا سكريبت العادية. هذا يقلل من التعقيد ويجعل الكود أسهل في التتبع.
    

#### **د. التصيير الديناميكي باستخدام (Array.map)**

- **ما هو؟** استخدام دالة `.map()` الخاصة بلغة جافا سكريبت لتحويل مصفوفة من البيانات (Objects) إلى مصفوفة من عناصر واجهة المستخدم (JSX Elements).
    
- **كيف يعمل؟ (مثال برمجي):**
    



```JavaScript
<select>
  {pizzaTypes.map((pizza) => (
    <option key={pizza.id} value={pizza.id}>
      {pizza.name}
    </option>
  ))}
</select>
```

### 3. الخطوات العملية (Step-by-Step Guide)

قام الشارح بربط نموذج الطلب (Order Form) بقاعدة بيانات حقيقية عبر API من خلال الخطوات التالية:

1. **تجهيز الحالة (State Setup):** إنشاء حالتين جديدتين؛ الأولى `pizzaTypes` (وتبدأ كمصفوفة فارغة `[]` لتخزين البيانات القادمة)، والثانية `loading` لمعرفة ما إذا كانت البيانات قيد التحميل.
    
2. **استنتاج البيانات (Derived State):** إنشاء متغيرات عادية `selectedPizza` و `price` للبحث في مصفوفة `pizzaTypes` بناءً على الـ ID الخاص بالبيتزا التي يختارها المستخدم.
    
3. **دالة الجلب (Fetch Function):** كتابة دالة `async` منفصلة باسم `fetchPizzaTypes` تستخدم `fetch` لجلب البيانات من الرابط `/api/pizzas`، ثم تحويلها إلى JSON، وحفظها باستخدام `setPizzaTypes`.
    
4. **تشغيل الـ Effect:** استدعاء الدالة السابقة داخل `useEffect` وإرفاق مصفوفة فارغة `[]` لضمان حدوث الجلب مرة واحدة فقط.
    
5. **التحديث الديناميكي للواجهة:** استبدال عناصر `<option>` الثابتة (Hardcoded) في القائمة المنسدلة بكود ديناميكي يمر على مصفوفة `pizzaTypes` باستخدام دالة `.map()`.
    
6. **تنسيق العملة:** استخدام واجهة المتصفح المدمجة `Intl.NumberFormat` لتنسيق السعر ليظهر كعملة رسمية (مثل الدولار USD).
    

### 4. الأخطاء الشائعة أو النصائح (Common Pitfalls & Tips)

- **الخطأ القاتل (Infinite Loop):** استدعاء الـ API بدون `useEffect` أو نسيان مصفوفة الاعتماديات `[]` سيجعل التطبيق يدخل في حلقة لا نهائية من التحديثات وطلبات الشبكة.
    
- **تحذير الدوال غير المتزامنة (Async inside useEffect):** لا تقم أبداً بكتابة `async` مباشرة على الدالة الممررة لـ `useEffect` (أي لا تكتب `useEffect(async () => {...})`). السبب هو أن الـ Async يُرجع دائماً كائن `Promise`، بينما يتوقع `useEffect` إرجاع دالة تنظيف (Cleanup Function) أو لا شيء. الحل هو تعريف دالة الـ Async بداخل أو بخارج الـ Effect ثم استدعاؤها.
    
- **نصيحة (Optimistic Rendering):** اكتب كود الواجهة وكأن البيانات (المصفوفة) موجودة بالفعل. هذا الأسلوب (التفكير التفاؤلي) يجعل الكود نظيفاً ويجنبك كتابة شروط معقدة (If/Else) غير ضرورية.
    

### 5. قاموس المصطلحات السريع

- **useEffect:** خطاف في React يُستخدم لتنفيذ أوامر أو تأثيرات جانبية تتزامن مع دورة حياة المكون.
    
- **Side Effect (تأثير جانبي):** أي عملية يقوم بها المكون للاتصال بخدمات خارجية أو التأثير على أنظمة خارج نطاق عودته المباشرة (مثل الطلبات الشبكية XHR).
    
- **Dependency Array (مصفوفة الاعتماديات):** مصفوفة تُمرر لـ `useEffect` لتحديد المتغيرات التي سيتم مراقبتها؛ وبناءً عليها يُقرر React متى يعيد تشغيل الـ Effect.
    
- **Derived State (الحالة المشتقة):** متغيرات يتم حساب قيمتها لحظياً بالاعتماد على بيانات `State` الأساسية، مما يلغي الحاجة لتخزينها في `State` منفصلة.
    
- **Cleanup Function (دالة التنظيف):** دالة يمكنك إرجاعها (Return) من داخل الـ `useEffect` لتنظيف التأثيرات السابقة (مثل إيقاف مؤقت زمني أو إلغاء طلب شبكي) قبل أن يُحذف المكون أو يُعاد تصييره.
    
- **Promise:** كائن برمجي في جافا سكريبت يمثل عملية لم تكتمل بعد (مثل انتظار استجابة الخادم)، وستُرجع إما نجاحاً (Resolve) أو فشلاً (Reject).

### 1. الخلاصة العامة (High-Level Summary)

يُسلط هذا المقطع الضوء على التحديات الشائعة عند التعامل مع الـ `useEffect` وجلب البيانات، ويحذر من خطأ برمجي شهير يسبب "حلقة لانهائية" (Infinite Loop) تؤدي لإغراق الخادم بالطلبات. كما يشرح كيفية التعامل مع بطء الشبكة ومعالجة أخطاء غياب البيانات قبل تحميلها باستخدام "التصيير المشروط" (Conditional Rendering) عبر المعامل الثلاثي (Ternary Operator).

### 2. المفاهيم والركائز الأساسية (Core Concepts)

#### **أ. الحلقة اللانهائية وإغراق الخادم (Infinite Loop & Self-DDoS)**

- **ما هو؟** خطأ برمجي يجعل المكون يعيد رسم نفسه (Re-render) ويستدعي الـ API آلاف المرات في ثوانٍ معدودة دون توقف.
    
- **لماذا يحدث؟ (المشكلة):** يحدث هذا عندما تقوم بتحديث الـ State داخل الـ `useEffect` ويكون هذا الـ Effect غير مضبوط بمصفوفة اعتماديات صحيحة. الدورة تكون كالتالي: تحديث الـ State ⬅️ يسبب Re-render ⬅️ الـ Re-render يُشغل الـ Effect ⬅️ الـ Effect يُحدث الـ State مرة أخرى ⬅️ وهكذا دواليك.
    
- **كيف نحله؟** يجب الانتباه الشديد لمصفوفة الاعتماديات `[]`. القاعدة الذهبية: إذا أشرت إلى متغير (State) داخل الـ Effect، فيجب وضعه غالباً داخل تلك المصفوفة، وإذا نسيت المصفوفة تماماً، فسيقع تطبيقك في هذا الفخ.
    

#### **ب. مشكلة غياب البيانات (Undefined Data Error)**

- **ما هو؟** انهيار مفاجئ للتطبيق عندما يحاول قراءة خصائص من كائن (Object) لم يتم جلبه من الخادم بعد.
    
- **لماذا نستخدمه؟ (المشكلة والحل):** في اللحظات الأولى من تشغيل التطبيق، تكون مصفوفة البيتزا فارغة، وبالتالي المتغير `selectedPizza` يكون `undefined`. إذا حاولت قراءة السعر أو الاسم منه، سيتوقف التطبيق (Crash). الحل هو منع React من عرض هذه المكونات حتى نتأكد من وصول البيانات.
    

#### **ج. التصيير المشروط (Conditional Rendering & Ternary Operator)**

- **ما هو؟** آلية لعرض أجزاء مختلفة من واجهة المستخدم بناءً على تحقق شرط معين (مثل: هل انتهى التحميل أم لا؟).
    
- **لماذا نستخدمه؟** لحل مشكلة البيانات غير المعرفة، ولتحسين تجربة المستخدم بعرض واجهة تحميل (Loading Spinner أو رسالة) بدلاً من شاشة بيضاء أو مكسورة.
    
- **كيف يعمل؟ (مثال برمجي):** يعتمد مطورو React بكثافة على "المعامل الثلاثي" (`? :`) داخل الـ JSX لأنه يُعتبر "تعبير برمجي" (Expression) يُرجع قيمة يمكن عرضها، وليس "جملة شرطية" (Statement) مثل `if/else` العادية.
    



```JavaScript
// إذا كان التطبيق قيد التحميل، اعرض رسالة، وإلا اعرض مكون البيتزا مع البيانات
{loading ? (
  <p>Loading pizza lol...</p>
) : (
  <Pizza 
    name={selectedPizza.name} 
    price={price} 
  />
)}
```

### 3. الخطوات العملية (Step-by-Step Guide)

تابع الشارح تطوير الصفحة عبر الخطوات التالية:

1. **محاكاة البطء (Artificial Delay):** أضاف الشارح تأخيراً برمجياً متعمداً لمدة 10 ثوانٍ (باستخدام `await` و `Promise`) داخل دالة الجلب ليرى بوضوح كيف يتصرف التطبيق قبل وصول البيانات.
    
2. **تمرير الخصائص (Props):** تم تمرير بيانات البيتزا المختارة (`name`, `description`, `image`) إلى مكون `<Pizza />` السفلي ليتم عرضها للمستخدم ديناميكياً.
    
3. **حساب السعر (Derived State):** تم استنتاج السعر برمجياً للبيتزا المختارة بناءً على الحجم الذي اختاره المستخدم `selectedPizza.sizes[pizzaSize]`.
    
4. **اكتشاف الخطأ (Debugging):** واجه الشارح خطأ `selectedPizza is undefined` لأن التطبيق حاول حساب السعر قبل أن يجلب الـ API البيانات، والسبب أن ميزة التحديث السريع (Hot Module Refresh) كانت تخفي الخطأ مؤقتاً في الذاكرة.
    
5. **تطبيق التصيير المشروط:** قام بتغليف المكون الشرطي بمعامل ثلاثي، بحيث يتم عرض رسالة `Loading pizza` طالما المتغير `loading` يساوي `true`. وبمجرد وصول البيانات وتغيره إلى `false`، يتم عرض البيتزا والسعر بدون أي أخطاء.
    

### 4. الأخطاء الشائعة أو النصائح (Common Pitfalls & Tips)

- **خطر تدمير الذات (DDoSing Yourself):** كن حذراً جداً عند تحديث أي `State` داخل الـ `useEffect`. نسيان الأقواس المربعة `[]` أو إدارتها بشكل خاطئ سيجعل تطبيقك يرسل طلبات API متتالية قد توقف السيرفر الخاص بك.
    
- **احذر من الـ Hot Module Refresh (HMR):** أدوات التطوير الحديثة رائعة وسريعة، لكنها قد تحتفظ ببيانات قديمة في الذاكرة (Memory) أثناء تعديل الكود. إذا كان كودك يعمل ثم انكسر فجأة دون سبب واضح، قم بعمل تحديث كامل للصفحة (Full Refresh) لترى الحالة الحقيقية لتطبيقك (كما حدث مع مشكلة الـ `undefined`).
    
- **لماذا المعامل الثلاثي؟** لا يمكنك كتابة عبارة `if ()` داخل أقواس `{ }` الخاصة بـ JSX. المعامل الثلاثي هو الحل الأنيق والقياسي في مجتمع React.
    

### 5. قاموس المصطلحات السريع

- **Infinite Loop (الحلقة اللانهائية):** دوران الكود في حلقة مفرغة من التنفيذ لا تتوقف أبداً بسبب خطأ منطقي في شروط الخروج أو الاستدعاء الذاتي.
    
- **Ternary Operator (المعامل الثلاثي):** صياغة مختصرة للجملة الشرطية في JavaScript تُكتب هكذا: `Condition ? True_Value : False_Value`.
    
- **Expression vs Statement:** التعبير (Expression) هو كود يُقيّم ويُنتج قيمة (مثل `5 + 5` أو المعامل الثلاثي)، بينما الجملة (Statement) هي تعليمة تنفذ أمراً ولا ترجع قيمة مباشرة (مثل `if` أو `for`). JSX يقبل الـ Expressions فقط بداخله.
    
- **Conditional Rendering (التصيير المشروط):** عملية رسم وعرض مكونات React بناءً على شروط منطقية متغيرة (مثل حالة تسجيل الدخول، أو حالة التحميل).
    
- **Hot Module Replacement (HMR):** ميزة في أدوات بناء التطبيقات تقوم بتحديث الكود المعدل في المتصفح فوراً دون الحاجة لإعادة تحميل (Refresh) الصفحة بالكامل.

### 1. الخلاصة العامة (High-Level Summary)

يُركز هذا المقطع القصير والهام جداً على خاصية الـ **`key`** في React عند توليد القوائم (Lists). يشرح الشارح آلية عمل هذه الخاصية وكيف تعتمد عليها React لإعادة ترتيب وتحديث عناصر واجهة المستخدم بكفاءة، مع تسليط الضوء على خطأ شائع جداً بين المطورين وهو استخدام رقم مؤشر المصفوفة (Index) كمفتاح، ومخاطر ذلك على استقرار التطبيق.

### 2. المفاهيم والركائز الأساسية (Core Concepts)

#### **أ. خاصية المفتاح (The `key` Prop)**

- **ما هو؟** سمة (Prop) خاصة وإجبارية في React تُمرر للعناصر التي يتم توليدها داخل حلقة تكرار (مثل استخدام دالة `.map()`)، وتُستخدم كـ "بطاقة هوية" فريدة لكل عنصر.
    
- **لماذا نستخدمه؟ (المشكلة والحل):** تخيل أن لديك قائمة معقدة من العناصر، وقام المستخدم بإعادة ترتيبها (مثلاً فلترة أو فرز). بدون وجود مفاتيح (`keys`)، لن تعرف React أي عنصر انتقل إلى أين. الحل الافتراضي لـ React في هذه الحالة سيكون "تدمير" القائمة القديمة بالكامل وإعادة بناء عناصر DOM جديدة من الصفر، وهو أمر مكلف جداً للأداء. بوجود الـ `key`، تتعرف React على العنصر وتقوم فقط "بنقله" إلى مكانه الجديد دون تدميره.
    
- **كيف يعمل؟** تقوم بتمرير قيمة فريدة ومرتبطة ببيانات العنصر نفسه (مثل `id` القادم من قاعدة البيانات) كـ `key` للعنصر الجذر.
    
- **مثال برمجي:**
    



```JavaScript
// الطريقة الصحيحة: استخدام مُعرّف فريد وثابت (ID)
{pizzaTypes.map((pizza) => (
  <option key={pizza.id} value={pizza.id}>
    {pizza.name}
  </option>
))}
```

### 3. الخطوات العملية (Step-by-Step Guide)

لم يقم الشارح ببناء ميزة جديدة هنا، بل قام بتجربة لتوضيح المفهوم:

1. **إزالة الـ Key:** قام بمسح خاصية `key` من الكود لملاحظة رد فعل React.
    
2. **رصد الخطأ:** أظهر المتصفح (في الـ Console) التحذير الشهير: `"Each child in a list should have a unique 'key' prop"`.
    
3. **تجربة الـ Index:** قام بتجربة استخدام متغير الـ `index` (ترتيب العنصر في المصفوفة) كبديل للمفتاح لشرح أسباب فشل هذه المقاربة في القوائم الديناميكية.
    

### 4. الأخطاء الشائعة أو النصائح (Common Pitfalls & Tips)

- **الفخ الأكبر (استخدام الـ Index كـ Key):** * **التحذير:** إياك واستخدام ترتيب العنصر في المصفوفة (Index 0, 1, 2...) كـ Key إذا كانت القائمة قابلة لإعادة الترتيب، الحذف، أو الإضافة.
    
    - **السبب منطقياً:** الـ Index غير مرتبط ببيانات العنصر نفسه (Not Stable). إذا كان لديك "بيتزا بيروني" في المؤشر `0`، وتم نقلها برمجياً إلى المؤشر `5`. بالنسبة لـ React، العنصر صاحب الـ Key `0` لم يتغير مكانه، لذا ستبقي React على واجهة المستخدم القديمة في مكانها، مما يؤدي إلى عرض بيانات خاطئة وانفصال الـ State عن الـ UI.
        
- **النصيحة الذهبية (الاستثناء):** متى يُسمح باستخدام الـ `index` كـ `key`؟ فقط وحصرياً إذا كانت القائمة "ثابتة تماماً" (Static)، أي أنك واثق بنسبة 100% أن عناصر القائمة لن يتغير ترتيبها أو يُحذف منها شيء أبدًا، والهدف الوحيد هو إسكات تحذيرات أداة فحص الكود (Linter).
    

### 5. قاموس المصطلحات السريع

- **Key Prop:** خاصية ضرورية في React تُستخدم كمعرف فريد ومستقر للعناصر المكررة لضمان تحديث الـ DOM بكفاءة.
    
- **Index:** رقم يُمثل ترتيب أو موقع العنصر داخل المصفوفة (يبدأ من 0)، وهو قيمة متغيرة وغير مرتبطة بهوية البيانات.
    
- **Linter / Linting:** أداة برمجية (مثل ESLint) تعمل في الخلفية أثناء كتابة الكود لتنبيه المطور للأخطاء المحتملة أو الممارسات غير القياسية (مثل نسيان كتابة خاصية الـ Key).
    
- **Stable Identifier:** مُعرّف ثابت لا يتغير بتغير مكان العنصر (مثل المعرفات الفريدة UUIDs أو Database IDs).

#### **ب. الوضع الصارم (Strict Mode)**

- **لماذا نستخدمه؟ (المشكلة والحل):** لاكتشاف الأخطاء الدقيقة والمخفية (مثل مشاكل الـ Side Effects)، ولتحذيرك من استخدام دوال أو ميزات سيتم حذفها من إصدارات React المستقبلية.

```JavaScript

```

#### **ج. أدوات مطوري رياكت (React DevTools)**

- **ما هو؟** إضافة للمتصفح (Extension) متوفرة لـ Chrome و Firefox توفر أدوات فحص مخصصة لـ React.
    
- **لماذا نستخدمه؟ (المشكلة والحل):** متصفح الويب العادي لا يفهم مكونات React أو الـ State، بل يرى فقط HTML. هذه الإضافة تترجم شجرة الـ DOM إلى شجرة مكونات React وتتيح لك رؤية ما يحدث خلف الكواليس.
    
- **كيف يعمل؟** بعد التثبيت، يضاف تبويبان جديدان لأدوات المطور:
    
    - **Components:** يعرض هيكل المكونات ويتيح لك تعديل الـ Props والـ Hooks (State) مباشرة لرؤية التغييرات فوراً.
        
    - **Profiler:** لتسجيل أداء التطبيق ورؤية ما يسبب البطء.
        

### 3. الخطوات العملية (Step-by-Step Guide)

تطبيق ما ورد في الفيديو لضبط بيئة العمل وتصحيح الأخطاء:

1. **تفعيل Strict Mode:** الذهاب إلى ملف البداية (غالباً `main.jsx` أو `app.jsx`) واستيراد `<StrictMode>` ثم تغليف الـ App به.
    
2. **تثبيت أدوات المطور:** تحميل إضافة React DevTools على المتصفح.
    
3. **تعديل الـ State حياً:** * فتح الـ Inspector في المتصفح، ثم الانتقال لتبويب `Components`.
    
    - الضغط على أي مكون (مثل `Order`).
        
    - ستظهر قائمة الـ Hooks على اليمين. يمكنك تعديل قيمة متغير (مثلاً تحويل الحجم من `S` إلى `L`) والضغط على Enter لترى الواجهة تتحدث فوراً دون تغيير الكود.
        

### 4. الأخطاء الشائعة أو النصائح (Common Pitfalls & Tips)

- **الخطأ الكارثي (فخ Slack):** تطبيق Slack الشهير قاموا بالخطأ برفع تطبيقهم للمستخدمين في وضع التطوير (Development Mode) لفترة طويلة. بمجرد أن قاموا بتغيير الـ `NODE_ENV` إلى `Production`، لاحظوا قفزة هائلة في الأداء. احرص دائماً على بناء التطبيق (Build) للإنتاج.
    
- **حيلة الـ Console (`$r` و `$0`):**
    
    - إذا حددت مكوناً من تبويب `Components`، يمكنك الذهاب لـ Console وكتابة `$r` لطباعة بيانات هذا المكون برمجياً.
        
    - وبالمثل لأي تقنية ويب، كتابة `$0` تطبع آخر عنصر HTML قمت بتحديده في تبويب `Elements`.
        
- **النصيحة الذهبية حول الأداء (Premature Optimization):** محرك React **"سريع كفاية"** بطبيعته لمعظم الاستخدامات العادية. لا تقلق بشأن الـ Commits و الـ Re-renders وتكتب أكواد معقدة لتحسين الأداء إلا إذا لاحظت بطئاً فعلياً في تطبيقك. حل المشاكل التي تواجهها بالفعل، ولا تحاول استباق مشاكل قد لا تحدث؛ لأن تحسينات الأداء تجعل الكود أصعب في القراءة والتعديل والصيانة.
    

### 5. قاموس المصطلحات السريع

- **NODE_ENV:** متغير بيئي يُحدد ما إذا كان التطبيق يعمل في بيئة تطوير (Development) لاختبار الكود، أو بيئة إنتاج (Production) لسرعة الأداء.
    
- **Strict Mode:** أداة فحص في React تكتشف الممارسات البرمجية السيئة وتُطلق تحذيرات لتجهيز كودك للتحديثات المستقبلية.
    
- **React DevTools:** إضافة للمتصفحات تُمكن المطور من التفاعل المباشر مع مكونات (Components) وحالة (State) تطبيق React من المتصفح مباشرة.
    
- **Flame Graph:** رسم بياني حراري داخل الـ Profiler يعرض الوقت المستغرق في تصيير كل مكون لمساعدتك على تحديد "عنق الزجاجة" (Bottleneck) في الأداء.
    
- **Commits:** المرحلة الأخيرة في دورة تحديث React، حيث تقوم React بتطبيق التغييرات المحسوبة فعلياً على الـ DOM الحقيقي ليراها المستخدم.
